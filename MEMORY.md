- **2026-05-10** — **Bankrun replay-protection workaround in tests: `setComputeUnitLimit({units: 200_001})` pre-instruction.** When two consecutive tests submit identical txs (same signer/accounts/args), Solana rejects the second as "transaction has already been processed" before the program runs. State-advancing transfers between tests don't reliably differentiate the txs. Adding a no-op `ComputeBudgetProgram.setComputeUnitLimit` pre-ix on the second tx changes its structure → different signature → no replay rejection. Used in `register_provider` duplicate-PDA + `accept_job` double-accept tests.
- **2026-05-10** — **Devnet program-metadata IDL refresh via `program-metadata write idl … --keypair`.** Anchor 1.0 CLI's `anchor idl init` and `anchor idl upgrade` both fail when the existing IDL metadata account is undersized for a larger new IDL (W1 IDL = 3.7KB, W2 IDL = 49KB). The standalone `@solana-program/program-metadata` CLI handles realloc — but **only when invoked with `--keypair ~/.config/solana/id.json` explicitly**; without it, the operation reports a generic "transaction plan failed" error. Documented for future redeploys.
- **2026-05-10** — **Worker inference path = mflux 0.17 / MLX subprocess on Apple Silicon, not diffusers/bitsandbytes/CUDA.** The dev box is an M3 Pro 18 GB so MLX-Metal is the native fast path; the project's PRD-stated "Windows 10/11 only at launch" applies to the **provider desktop installer**, not the developer workstation that runs the worker. The W3+ Linux/CUDA path stays open via a future `inference_backend` switch, but for hackathon scope the single-backend mflux is enough. **mflux 0.17 has no stable Python API** — the package was restructured and now only exposes CLI commands (`mflux-generate`, `mflux-controlnet`, etc.). `apis_worker/inference.py` is therefore a `subprocess.run([sys.executable.parent / "mflux-generate", ...])` wrapper. Defaults: 4 steps, 8-bit quant, 1024×1024 (override per-spec). First run is slow (kernels JIT-compile on first dispatch — 3-4 min for a 512² job) but warm runs land in ~50s on M3 Pro 18 GB.
- **2026-05-10** — **Pinata v3 API, not the legacy v1 `pinFileToIPFS`.** Pinata reorganized the dashboard/API in 2024-25; the new key UI exposes `Files / Groups / Gateways / Analytics` scopes (None/Read/Write) instead of legacy IPFS-pinning scopes. Worker uses the v3 endpoint `POST https://uploads.pinata.cloud/v3/files` (multipart, `Authorization: Bearer <JWT>`) with `Files: Write` scope. Response shape is `{data: {cid, ...}}` — note the wrapping `data` key, distinct from v1's flat shape. `apis_worker/ipfs.py` lives behind a thin `httpx.AsyncClient`.
- **2026-05-10** — **Buyer→worker spec side-channel = filesystem stub at `/tmp/apis_specs/{spec_hash_hex}.json`.** On-chain Job stores only `spec_hash` (32 bytes); the actual prompt+model+steps+dimensions JSON has to reach the worker out-of-band. For W2 we ship a file-based stub (`apis_worker/spec_channel.py`: `store_spec()`, `lookup_spec()`) so the buyer-side test scripts can drop the spec where the worker can read it. **W4 replaces this with the MCP/x402 server**, which will be the real prompt-delivery channel. The stub uses canonical-JSON encoding (`sort_keys=True, separators=(",", ":")`) so the buyer's spec_hash and worker's spec verification both round-trip identically.
- **2026-05-10** — **`TxOpts(preflight_commitment=Confirmed)` is mandatory for worker tx sends.** Default preflight commitment is `Processed` (tip-of-leader), which means the worker's `accept_job` simulation runs against a slot that may not yet contain the buyer's just-confirmed `create_job`. Symptom: `accept_job` returns `Custom(3012)` / `0xbc4` ("AccountNotInitialized") on the Job PDA even though `confirm_transaction(create_job_sig, "confirmed")` already returned. ## 🐛 Known Issues & Quirks

*(Log current bugs or weird workarounds here.)*

- **Anchor 1.0.2 vs older training data:** Most Anchor tutorials and Claude's training data lean on 0.30.x. Several APIs and macros changed in 1.0. **Mitigation:** every prompt to Claude must include *"Per Anchor 1.0.2 docs at anchor-lang.com — note 1.0.x has breaking changes vs 0.30.x"*. Cross-reference recent commits in `solana-developers/program-examples`.
- **Phantom desktop has no real deeplink protocol** (unlike mobile). For the Tauri provider app, do not use Phantom — use **Privy embedded wallet** OR a local Stronghold-encrypted keypair (Tech Design §4 F1).
- **Flux.1 Schnell requires `guidance_scale = 0`.** It's a distilled model — non-zero CFG produces garbage. Hard-code in worker. `max_sequence_length` ≤ 256.
- **Tauri 2 permissions system** is stricter than v1 — every plugin command needs an explicit capability JSON entry. Budget ~1 day for the permission wiring during W4.
- **Don't bundle PyTorch + CUDA** in the worker installer (~6-8 GB → AV false positives + bricked installs). Ship a thin installer + first-run model downloader with hash verification (Tech Design §13).
- **EV code signing requires a hardware token** (post-Jun 2023 rule) — defer to Phase 2 (~$200/yr). At hackathon: live with SmartScreen warnings; demo provider runs from dev machine.
- **Solana devnet can lag** during high-traffic periods. **Demo strategy:** pre-record the demo video; do not rely on a live demo during the 3-min hackathon submission (Tech Design §13 risk #2).
- **Solana CLI 3.1.15 is not in AVM's release archive** (only the system installer has it). Anchor 1.0.2 falls back to the AVM-cached **3.1.8** when `[toolchain] solana_version = "3.1.15"` is set. **Mitigation:** pin `solana_version = "3.1.8"` in `Anchor.toml` until AVM publishes 3.1.15. Migration guide §0 recommends 3.1.10+, but 3.1.8 works fine with Anchor 1.0.2 in practice (verified end-to-end: build, test, devnet deploy).
- **`packages/program` is excluded from the pnpm workspace** (per `pnpm-workspace.yaml` comment — "Anchor manages its own JS deps"). Running `pnpm install` from inside the package without `--ignore-workspace` is a no-op. **Use `pnpm install --ignore-workspace`** in `packages/program/` to install the test runner / Anchor TS client deps.
- **Root `.gitignore` had a too-broad `lib/` rule** (Python virtualenv convention) that masked `packages/web/app/lib/`. Anchored to `/lib/` and `/lib64/` (repo root only) on 2026-05-10 — the framework's `app/lib/` and similar dirs are tracked again.
- **Anchor 1.0 program upgrades require pre-extension when the new binary is bigger than the on-chain ProgramData allocation.** `anchor program deploy` errors with "ProgramData account not large enough"; fix is `solana program extend <PROGRAM_ID> <ADDITIONAL_BYTES>` first. Also, the upgrade allocates a temporary write-buffer account whose rent (~$2.24 SOL for our 321KB binary) must be paid up front by the deployer; rent is refunded after the swap. Devnet faucet rate limits (2 airdrops / 8h / IP) can leave the deployer wallet under-funded mid-upgrade — workaround = top up from Phantom (devnet) via in-wallet "Send" to the deployer pubkey.
- **Anchor 1.0 `anchor idl init`/`upgrade` fail on size-grew IDL accounts.** Use `npm install -g @solana-program/program-metadata` then `program-metadata write idl <PROGRAM_ID> target/idl/apis_program.json --rpc <RPC> --keypair ~/.config/solana/id.json` instead. The `--keypair` flag is mandatory — without it the CLI reports a generic "transaction plan failed" error.
- ~~**Worker WebSocket has no auto-reconnect.**~~ Fixed in W2-Step-3.5 — `listen_for_jobs` wraps the connect-and-subscribe block in an exponential-backoff retry loop catching `ConnectionClosed*`, `IncompleteReadError`, `OSError`. Reconnects after 1s → 2s → 4s … capped at 30s; resets on successful subscription. Inflight `_process_job` tasks unaffected (they use HTTP-RPC, not the WS sub).
- **mflux 0.17 first-run is slow (~3-4 min for 512²).** MLX kernels JIT-compile on first dispatch per (resolution, dtype, quant) tuple; subsequent runs at the same shape land in ~50s on M3 Pro 18 GB. **Mitigation for demo:** pre-warm the kernels by running one throwaway `mflux-generate` at the demo resolution before the live recording. Cache lives at `~/.cache/huggingface/hub/models--black-forest-labs--FLUX.1-schnell` (~31 GB after first download — multiple checkpoint formats are pulled).
- ~~**Concurrent mflux subprocesses on M3 Pro 18 GB ≈ ~5× slowdown each**~~ Fixed in Sprint 1.4 — `apis_worker/listener._job_lock = asyncio.Lock()` serializes the full pipeline (accept_job → inference → upload → submit_completion). Concurrent JobCreated tasks now queue in arrival order. Test at `packages/worker/tests/test_listener_lock.py`.
- **`spl-token create-account` is not idempotent in the bundled CLI version** (no `-i` / `--idempotent` flag). Repeat invocations fail with `Error: Account already exists: <ATA>` on stderr + exit-1. `bootstrap_devnet.py` now has an `_ensure_ata_exists()` helper that catches `CalledProcessError`, looks for "already exists" in stderr, and parses the ATA address out of either the success or error output. The script is fully idempotent across re-runs.
- **HuggingFace FLUX.1-schnell is a gated repo.** Requires a logged-in HF account that has clicked "Agree and access repository" on https://huggingface.co/black-forest-labs/FLUX.1-schnell **and** an `HF_TOKEN` (Read scope) in `packages/worker/.env`. First-time worker setup error if missing: `403 Forbidden: User not in authorized list`. Documented in worker README (TODO).
